## Introduction

### Objective

Detect or predict ransomware activity based on the way
data is updated (encrypted, obfuscated, shuffled, network communication
to C2 servers, etc.) with malicious intent (ransom). By analyzing the
requests to update the contents in the production data, we aim to
develop a detection system that can identify, predict, and explain potential
ransomware activity before it causes significant damage.

For instance, instead of just looking for encryption after it happens, the
system might analyze the pattern of requests made to the data (e.g., sequences
of file writes, access patterns) to detect suspicious activity, such as
gradual data encryption (salami-slice method), obfuscation, or shuffling, that
might not be immediately recognized as ransomware behavior using traditional
detection methods.

## Challenges

### Granularity of Detection

The research requires more fine-grained monitoring of data update requests,
which is challenging because ransomware may employ methods that disguise their
true intentions (e.g., gradually manipulating the data).

### Data Abundance and Complexity

Ransomware detection requires analyzing a wide range of data sources, including
file system operations, network traffic, and process activities. The sheer volume
and complexity of this data can make it challenging to identify subtle patterns
of malicious activity. This is a big data problem. If the ransomeware activity
is "dense" in the data, then detecting it is more likely to be successful. However,
in real data, the ransomware activity may be very sparse, and the detection
system may need to be able to see the needle in the haystack. Long-context
language models are becoming increasingly effective at this kind of task, but
they are also very computationally expensive. If we generate synthetic data,
we can parameterize the sparsity of the ransomware activity and see how well
the model can detect it for the given sparsity.

### Proactive Detection

The goal is to detect malicious activity as it happens, rather than reacting
after the damage is done. This requires sophisticated approaches that can discern malicious intent from legitimate data operations in near real-time, which can be costly in terms of computational resources and complexity.

### Negligible False Negative Rate

We aim to achieve a false negative rate close to $0$, meaning the system should
not be expected to miss many malicious updates. This is a may be a challenge,
especially with the evolution of ransomware techniques like the salami-slice method
or obfuscation that can evade traditional detection methods.

### Tolerable False Positives:

While striving for a neglible false negative rate, the system must also tolerate
a reasonably small false positive rate. This balance is critical since excessive
false positives can reduce the usability of the system, leading to alarm fatigue.

## Possible Approach: Sequence Prediction on from Multiple Sources

Given the sophistication of ransomware techniques, a multi-source data collection
approach may be necessary to detect subtle patterns of malicious activity. Since
the ransomeware activity can be obfuscated and spread out over time, a sequence
prediction model that learns from multiple sources of data (e.g., file system
operations, process activities, network traffic) may be appropriate.

predict these explanations (it will insert its predictions into the sequence)

**Assumption:** The system operates in an ideal research environment with no computational or resource constraints. The focus is on developing and testing the effectiveness of the detection methods without the limitations of real-time production systems.

### Tools and Setup

A serious decision to make is what tools to use to collect training data. I have
identified two general approaches:

1. Existing collections of ransomware activity collected from real-world attacks.
2. Synthetic ransomware activity generated in a controlled environment.

For (1), the advantage is that the data is real and can be used to train a model
for real-world scenarios. The disadvantage is that the data is not as controlled
and may not be as rich as we would like. For (2), the advantage is that we can
control the data and generate as much as we need. The disadvantage is that the
data may not be as realistic as real-world data.

### Synthtic Ransomware Data

There are two general approaches to generating synthetic ransomware data. The
first approach is to run actual ransomware in a controlled networked VM environment.
The second approach is to use a ransomware emulator. The advantage of the emulator
is that it is easier to control and to generate data, but the disadvantage is that it may not be as realistic or as rich as running actual ransomware programs in our controlled environment.

### Data Collection and Preprocessing

- **Comprehensive Data Collection:**
    - **Auditd Logs:** Captures detailed file I/O operations and process activities.
    - **Strace Logs:** Captures system call traces, especially file and network-related calls.
    - **Suricata Alerts:** Captures network traffic anomalies, such as C2 communication, unusual DNS queries, and large outbound traffic spikes.
- **Synthetic Ransomware Activity:** Run simulated ransomware programs to generate realistic attack data (e.g., salami-slice method). Augment the logs with metadata or annotations about known ransomware activity.
- **Feature Extraction:** We want to minimize handcrafted feature engineering.
We are taking Rich Sutton's "The Bitter Lesson" to heart. We will use deep learning to learn the features from the raw data. However, we may need to extract some features from the raw data to make it more amenable to deep learning. For example, we may need to convert system calls into a more structured format to feed into the model.

### Ransomware Detection Approach

- **Sequence Prediction Model:**
    - **Model Type:** Sequence prediction model (e.g., Transformer) that learns temporal and sequential patterns from a multi-source dataset (file system operations, process activities, and network traffic).
    - **Training Data:** Use labeled sequences from the combined dataset, augmented with metadata indicating whether the sequences represent normal or ransomware activity.
    - **Evaluation:** Evaluate the model’s ability to detect ransomware based on the full dataset from multiple sources, assessing for both false positives and false negatives.

- **Ideal Environment for Model Training:** In this research setup, there are no constraints on computational power or resource usage, allowing for deep exploration of model architectures, training techniques, and hyperparameter tuning.

### Research Hypothesis
   - **Hypothesis:** By leveraging a comprehensive dataset that includes file system operations, process activities, and network traffic, a sequence prediction model can effectively detect a wide range of ransomware activities. The integration of multi-source data enables the model to learn subtle patterns of malicious behavior and distinguish them from normal operations, and
   provide predictions and explanations for potential ransomware activity that may be missed by traditional detection methods. These predictions can be used to trigger alerts or other actions to prevent data loss or corruption before it occurs.
